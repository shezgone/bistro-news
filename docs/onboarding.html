뉴스 기반 인과관계 분석 — 거시경제 모델용 · 2026-06-17 · 기획팀·개발팀 공동

한 줄 목적. 뉴스 기사에서 경제 사건을 뽑아 "무엇이 무엇을 일으켰나"를 그래프로 엮고, 이를 숫자(지수)로 압축해 거시경제 예측 모델의 입력 신호로 쓴다.

1. 파이프라인 한눈에

뉴스 수집 → 중복제거 → 이벤트 추출(LLM) → 클러스터링 → 인과 그래프 → 지수화 → 거시모델
 (BIGKinds)  (SHA256+    (Step A/B/C/D)               (3단계 필터)  (Track A/B)  (Granger 검증)
             BGE-M3)
단계하는 일
1. 수집·중복제거기사 수집 → SHA256(동일) + BGE-M3 임베딩(유사) 클러스터링, cluster_uuid 관리
2. 이벤트 추출LLM 4-step: A Core → B Background → C Consequence → D Quality Critic
3. 클러스터링추출된 이벤트를 임베딩 유사도로 동일 계열끼리 묶음 (기사 경계 너머 연결)
4. 인과 그래프시간방향 강제 → 가중치(선행성×유사도×보도빈도) → LLM+PCMCI 이중검증
5. 지수화Track A(보도강도) / Track B(eigenvector centrality 변화량) → 일별 지수
6. 검증 루프Granger 인과검정 + lead-lag 구조 → 역방향으로 스펙 재조정

근거 논문: Tilly & Livan (UCL), 2021, Macroeconomic forecasting with statistically validated knowledge graphs, arXiv:2104.10457.

2. 전체 흐름 예시 (실제 테스트 기사)

입력 기사 3건: ① 호르무즈 통행료(한국경제) ② 같은 호르무즈 보도(타 언론사) ③ 범정부 교통TF 발족(연합뉴스)

단계이 예시에서 일어나는 일
① 중복제거SHA256은 ①②를 못 거름(문구 다름) → BGE-M3 유사도 0.93으로 같은 cluster_uuid. 대표 1건만 추출(비용↓) + "2개사 보도" 기록
② 추출기사마다 작은 미니그래프(배경→Core→결과) 생성. 아직 서로 끊김
③ 클러스터링 ★기사①의 결과 "고유가" ↔ 기사③의 배경 "중동발 고유가" → 같은 노드로 연결
④ 그래프끊긴 미니그래프들이 ③으로 이어져 아래 인과 사슬 완성
⑤ 지수화"고유가" 노드가 여러 사슬의 허브 → centrality 높음 → 그날 지수에 크게 기여
⑥ 검증이 지수가 실제 ECOS 에너지·물가 통계에 선행하는지 Granger 검정
그림 1. 두 기사가 공유 노드(★)를 통해 하나의 인과 사슬로 연결된 모습 (Graphviz 렌더링)

3. 클러스터링을 왜 하나? (사실 2번 한다)

단계무엇을 묶나왜
① 기사 중복제거같은 사건 기사들중복 기사를 1건으로 → 추출 비용 절감 + 보도 강도 정확 측정
③ 이벤트 클러스터링다른 기사의 같은 이벤트아래 3가지 (핵심)
③ 이벤트 클러스터링이 없으면 무너지는 것
  1. 그래프가 조각남 — 같은 "고유가"가 기사마다 다른 노드가 돼 미니그래프가 영영 안 이어짐. 클러스터링이 기사 경계를 꿰매 하나의 인과 네트워크로 만듦.
  2. 중요도(보도빈도) 측정 불가 — "30개 기사가 보도 = 중요" 신호는 30건을 같은 노드로 묶어야 셀 수 있음 → 가중치·Track A의 재료.
  3. 예측→실현 추적 불가 — 한 기사의 예측된 결과가 다른 기사의 실제 사건과 같은 것임을 알아야 "예측이 맞았나"를 검증(PCMCI).
요약: 추출은 기사별 조각난 미니그래프만 만든다. 클러스터링이 그 조각을 이어붙여 "중요도 가중치 + 기사 경계 넘는 인과 사슬"을 가진 하나의 그래프로 만든다.

4. 이거 경제 온톨로지 아닌가? 개념 정리

맞다 — 다만 정확히는 "얇은 온톨로지 층을 가진 이벤트 지식그래프(event knowledge graph)"다. 이 구분이 설계에 중요하다.

층온톨로지?이 프로젝트에서
클래스/타입✅event_type enum, entities(organizations/persons…) = 분류체계
관계 타입✅relation_to_core(CAUSE…), consequence_type = object property
스키마✅event_schema.json = 경량 온톨로지(클래스+속성 정의)
인과 엣지(사실)❌"4/7 이 휴전이 → 이 고유가를" = 인스턴스 레벨 · 시점 있음 · 확률적(weight·PCMCI)

핵심 차이. 전통 온톨로지(OWL/RDF)는 타입 레벨·시간 무관·논리적으로 확실한 관계를 다룬다. 이 프로젝트의 인과 엣지는 토큰 레벨·시점 있음·통계적으로 검증되는 주장이다. → 스키마 층은 온톨로지, 인과 층은 경험적 그래프.

⚠️ 함정. 이걸 "경제 온톨로지"로 통째로 보고 인과 층까지 OWL/추론기로 형식화하면 안 된다. 거시 인과는 확률적·시변(time-varying)이라 형식 온톨로지가 가장 못 다루는 영역. PCMCI·LLM critic 같은 경험적 검증이 맞다.

온톨로지 프레이밍이 실제로 이득인 지점

  • 엔티티 층 — 기존 표준 재사용 + 엔티티 링킹. organizations·persons·countries를 Wikidata QID로 링킹하면(예: 美 연준 = Q53536) 기사마다 "Fed/연준/연방준비제도"가 다른 노드가 되는 걸 막아 centrality 시계열이 안정되고 클러스터링 부담도 줄어든다. 금융 분류는 FIBO, 이벤트 코딩은 CAMEO(GDELT) 검토.
  • 저장. 형식추론이 필요 없으니 RDF 트리플스토어보다 property graph(Neo4j)가 가중·시점 엣지에 더 맞다.

6. 구현 전 합의할 설계 포인트

  1. 노드 정의 — 이벤트 노드 vs 엔티티/event_type 레이어. centrality 안정성과 직결. (Wikidata 링킹 권장)
  2. 예측↔실현 병합 규칙 — 기사 A의 consequence(예측)와 기사 B의 core(사실)가 매칭될 때: (a) 단일 노드 병합 vs (b) 별도 노드 + REALIZED_AS 엣지(권장, lead-lag 추적 유리).
  3. baseline부터 이기기 — 단순 키워드 지수 + Track A를 먼저 만들고, 복잡한 인과그래프 Track B가 실제로 능가하는지 확인.
  4. 추출 정량평가 — 현재 기사 2건 정성평가뿐. 필드 정확도·core 1개 준수율·is_factual 오류율 벤치마크 필요.
