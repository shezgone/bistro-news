# BISTRO-News 온보딩 (기획팀)

> **이 문서는 무엇인가**: 기획팀이 이 레포에 처음 들어와 *어디부터 보고, 무엇이 내 영역이며, 어떻게 기여하는지* 5분 안에 파악하기 위한 진입점.
> 이 레포는 **Claude Code에서 열어 질문하며 탐색**하도록 세팅돼 있습니다. 모르는 건 그냥 물어보세요 (아래 "Claude Code로 묻는 법").

## 내 영역은 어디인가 (기획팀 주 작업 구역)

| 디렉터리 | 내용 | 기획팀 권한 |
|---|---|---|
| [prompts/](prompts/) | 이벤트 추출 프롬프트 — `step_a_core` / `step_b_background` / `step_c_consequence` / `step_d_critic` (+ `legacy_single`) | **주 작업** ✍️ |
| [schemas/event_schema.json](schemas/event_schema.json) | 추출 결과 JSON **계약(contract)** — 필드·enum 정의 | **합의 후 변경** ⚠️ |

| [docs/](docs/) | 회의·중간 요약·스펙 | 날짜 붙여 보관 |

> `src/bistro_news/`(파이프라인 코드)는 개발팀 영역. 현재 구현된 건 `extract`(평가 경로)뿐이고 나머지 단계는 스텁입니다.

## 기여하는 법

1. **프롬프트 개선** → `prompts/`의 해당 단계 파일 수정. 변경 **이유**를 커밋 메시지에 남긴다 (예: "Step A: 익명 출처 is_factual=false 규칙 강화").
2. **출력 스키마 변경** → `schemas/event_schema.json`이 추출 결과의 계약. 필드/enum 추가·변경은 **개발팀과 먼저 합의** 후 이 파일을 수정 (깨지면 평가·다운스트림이 전부 영향).
3. **설계·방법론 결정** → `CLAUDE.md`에 반영. 큰 결정은 여기, 회의록은 `docs/`에 날짜 붙여서.
4. **브랜치 전략** → `main` 직접 푸시 ❌. 기능 브랜치 + **PR** → 리뷰로 스키마 호환성 확인.

## Claude Code로 묻는 법 (이 레포 전용)

이 레포를 Claude Code에서 열면 `CLAUDE.md`를 자동 인지합니다. 예시 질문:

- *"신규 4-step 프롬프트가 기존 단일 프롬프트와 뭐가 다른지 단계별로 설명해줘"*
- *"Step A에서 is_factual 판별 규칙을 보여주고, 어떻게 바꾸면 익명 보도를 더 잘 걸러낼까?"*
- *"event_schema.json에 필드 하나 추가하면 eval 어디가 깨지나?"*
- *"지금 구현된 코드와 아직 스텁인 단계를 구분해줘"*

> 코드를 직접 안 짜도 됩니다. "이 부분 왜 이렇게 설계했어?" 류 질문이 가장 효율적이에요.
